# Real Data Quality Assessment

## Dataset Status

The assessment pipeline is implemented in:

```bash
python analysis/evaluation/run_quality_assessment.py --write
```

For a safe demo run without real collector output:

```bash
python analysis/evaluation/run_quality_assessment.py --demo --write
```

Default input directories:

- `collectors/processed_storage`
- `collectors/raw_storage`

These collector output directories are intentionally git-ignored. The current
repository snapshot does not include real collected documents, so final Sprint 2
calibration must be re-run after the local collector dataset contains at least
200 parsed documents.

The committed fallback dataset in `docs/analysis/synthetic_quality_dataset` is
synthetic and non-sensitive. It exists only to validate the quality assessment
pipeline before real data is available.

## Required Metrics

The generated report calculates:

- `leak_quality_score = relevant_documents / total_documents`
- Total documents analyzed
- Relevant vs irrelevant classification counts
- Spam/noise document count
- Duplicate document count
- Language distribution
- Pattern counts by pattern type
- Terminology hit counts
- Terminology false-positive candidates
- Classification accuracy when labels are available
- Average confidence per detected pattern type
- Top 5-10 suspicious or high-risk real finding candidates

## Calibration Rules

The model tuning pass uses these decisions:

| Signal | Calibration decision |
|---|---|
| Dataset smaller than 200 documents | Do not finalize thresholds; collect more data |
| Synthetic demo dataset used | Mark results as demo-only; do not claim real-world calibration |
| Low leak quality score below 0.30 | Keep escalation threshold at 80+ |
| Duplicate documents detected | Review deduplication before scoring calibration |
| Parser spam/noise detected | Tune parser noise filter before final model tuning |
| Terminology hits on irrelevant docs | Down-weight ambiguous terms and review context filters |

## Pattern Validation

Pattern validation is measured by counting detector outputs across the collected
documents. When labeled samples are available, the existing pattern test runner
provides precision and recall:

```bash
python analysis/evaluation/run_pattern_tests.py
```

Targets remain:

- Precision: at least 80 percent
- Recall: at least 70 percent
- High-risk recall: at least 90 percent
- False-positive rate: at most 20 percent

## Terminology Accuracy

Terminology accuracy is reviewed with two signals:

- Total term hits by term.
- Documents where terminology appears but the final classification remains
  `irrelevant`.

Those irrelevant-document hits are treated as false-positive candidates for
manual review. Ambiguous terms such as `log`, `secret`, and `exclusive` should
remain conditional and low-weight unless supported by stronger evidence.

## Classification Calibration

Classification calibration compares `expected_classification` metadata with the
analysis engine output when labels are available in JSON source documents.

Supported labels:

- `irrelevant`
- `related`
- `suspicious`
- `high-risk`

If labels are not present, the report still provides class distribution and top
candidate findings, but it marks accuracy as unavailable.

## Confidence Thresholds

Pattern confidence is summarized per pattern type. Current tuning rule:

- Keep high-confidence credential, private key, and API key patterns unchanged.
- Review patterns with average confidence below `0.85`.
- Treat low-confidence config secrets as supporting evidence, not standalone
  escalation evidence.

## False Positive Handling Strategy

- Keep analyst review as the final gate before escalation.
- Prefer score thresholds of `80+` for analyst escalation when collected data is
  noise-heavy.
- Keep `is_false_positive` and `review_notes` as feedback signals for future
  model tuning.
- Re-run quality assessment after collector changes, terminology changes, or
  scoring formula changes.
